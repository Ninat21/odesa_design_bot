# Architecture Decisions

Цей документ містить усі важливі архітектурні рішення проєкту.

Мета документа — пояснити, чому було прийнято певне рішення, щоб у майбутньому не довелося повторно аналізувати одні й ті самі питання.

---

# ADR-001

## Назва

PostgreSQL як єдине джерело даних.

## Статус

Прийнято.

## Причина

Усі модулі повинні працювати з однією базою даних.

Не допускається дублювання даних між сервісами.

---

# ADR-002

## Назва

SQLAlchemy як ORM.

## Статус

Прийнято.

## Причина

- підтримка PostgreSQL;
- Alembic;
- хороша документація;
- зріла екосистема.

# ADR-003

## Назва

Alembic для міграцій.

## Статус

Прийнято.

## Причина

Будь-які зміни структури бази повинні виконуватися лише через міграції.

Ручна зміна структури бази заборонена.

# ADR-004

## Назва

Telegram є інтерфейсом, а не ядром системи.

## Статус

Прийнято.

## Причина

Архітектура не повинна залежати від конкретного месенджера.

---

# ADR-005

## Назва

AI не змінює фактичні дані.

## Статус

Прийнято.

## Причина

AI може помилятися.

Фактичні дані повинні залишатися достовірними незалежно від роботи AI.

# ADR-014

## Назва

Файли не зберігаються в базі даних.

## Статус

Прийнято.

## Причина

Платформа використовує Telegram як основне сховище медіафайлів.

У базі даних зберігаються лише метадані та Telegram File ID.

Це дозволяє:

- значно зменшити розмір бази даних;
- спростити резервне копіювання;
- уникнути дублювання даних;
- не витрачати місце на файловому сховищі.

У разі потреби файл може бути повторно отриманий через Telegram Bot API.

У майбутньому допускається додавання локального кешу або зовнішнього файлового сховища без зміни структури бази даних.

# Правила внесення нових рішень

Кожне нове архітектурне рішення повинно містити:

- номер;
- назву;
- статус;
- дату;
- причину;
- можливі альтернативи;
- наслідки прийнятого рішення.

Після прийняття рішення всі пов'язані документи повинні бути оновлені.
