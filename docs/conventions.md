# Conventions

Цей документ визначає єдині правила розробки проєкту.

Дотримання цих правил є обов'язковим для всіх модулів системи.

---

# Загальні принципи

## Простота

Будь-яке рішення повинно бути максимально простим.

Не допускається ускладнення архітектури без реальної потреби.

---

## Єдиний стиль

Усі модулі повинні бути написані в одному стилі.

Новий код має бути схожим на вже існуючий.

---

## Читабельність

Код читається значно частіше, ніж пишеться.

Пріоритет віддається зрозумілості.

---

# Мова

## Код

Уся програмна логіка пишеться англійською мовою.

До коду належать:

- назви файлів;
- назви класів;
- назви функцій;
- назви змінних;
- назви таблиць;
- назви колонок;
- ENUM;
- константи.

---

## Документація

Уся внутрішня документація ведеться українською мовою.

---

## Коментарі

Коментарі в коді пишуться українською мовою.

Коментарі використовуються лише тоді, коли код неможливо зробити самодокументованим.

---

# Структура моделей

Порядок елементів у кожній SQLAlchemy-моделі:

1. __tablename__
2. __table_args__
3. Primary Key
4. Foreign Keys
5. Основні поля
6. JSON / JSONB поля
7. Relationships

Усі моделі повинні мати однакову структуру.

---

# Repository Pattern

Handler не працює напряму з базою даних.

Правильний ланцюжок:

```
Handler

↓

Service

↓

Repository

↓

Database
```

---

# Бізнес-логіка

Бізнес-логіка розміщується лише у Services.

Repository не містить бізнес-логіки.

---

# Форматування

Використовуються:

- Black
- Ruff

Форматування вручну не виконується.

---

# Іменування

## Класи

```
UserRepository

AnalyticsService

ImportService
```

---

## Функції

```
get_user()

create_profile()

update_message()
```

snake_case.

---

## Константи

```
MAX_MESSAGE_LENGTH

DEFAULT_PAGE_SIZE
```

UPPER_CASE.

---

## ENUM

Назви ENUM:

```
MessageType

EventStatus

NotificationStatus
```

Значення ENUM:

```
TEXT

PHOTO

VIDEO
```

---

# Дати

Усі дати зберігаються у UTC.

Локальний час визначається лише під час відображення.

---

# Ідентифікатори

Первинний ключ:

```
id
```

Зовнішній ключ:

```
user_id

event_id

project_id
```

Не використовуються скорочення.

---

# Логування

Використовується стандартний модуль logging.

Логи повинні бути інформативними.

Не допускається використання print() у робочому коді.

---

# Обробка помилок

Помилки не повинні приховуватися.

Усі винятки або обробляються, або логуються.

---

# Документація

Кожна нова сутність повинна бути описана у відповідному документі.

Якщо змінюється архітектура, необхідно оновити:

- architecture.md
- database_design.md
- decisions.md

---

# Git

Один логічний коміт — одна завершена зміна.

Не допускається змішування різних задач в одному коміті.

---

# Тестування

Новий функціонал повинен перевірятися до інтеграції.

Критично важлива логіка повинна мати автоматичні тести.

---

# Головне правило

Якщо існує декілька варіантів реалізації, перевага надається тому, який:

- простіший;
- зрозуміліший;
- легше підтримується;
- відповідає існуючій архітектурі.
