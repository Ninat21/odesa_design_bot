# Общие принципы

База данных проектируется как основа платформы управления профессиональными сообществами.

Основные требования:

- масштабируемость;
- расширяемость;
- сохранение истории;
- независимость от Telegram;
- поддержка нескольких сообществ;
- поддержка нескольких источников данных;
- совместимость с AI.

---

# Правила проектирования

## Пользователь не зависит от Telegram

Внутренний идентификатор пользователя никогда не совпадает с Telegram ID.

Telegram является лишь одним из способов идентификации пользователя.

---

## История важнее состояния

Изменения не удаляются.

Любая информация может быть восстановлена.

---

## AI не изменяет факты

AI хранит собственные данные отдельно.

Фактические данные принадлежат только соответствующим сущностям.

---

## Любое изменение имеет источник

Например:

- Telegram
- Telethon
- Luma
- Website
- Admin
- AI
- Import

---

## Все даты хранятся в UTC.

---

## Все сущности имеют

- created_at
- updated_at

# Связи

Все связи должны быть явными.

Используются:

- One-To-One
- One-To-Many
- Many-To-Many

Не допускается хранение списков идентификаторов в JSON.

---

# JSONB

JSONB используется только там, где невозможно заранее определить структуру.

Например:

- AI Summary
- AI Preferences
- AI Communication
- AI Career

JSONB не используется вместо нормализованных справочников.

---

# ENUM

Для повторяющихся значений используются ENUM.

Например:

- MessageType
- EventStatus
- TimelineEvent
- NotificationStatus
- ContactType

---

# Индексы

Индексы создаются только для полей, которые используются при:

- поиске;
- сортировке;
- фильтрации;
- JOIN.

Избыточные индексы не допускаются.

---

# Внешние ключи

Все связи реализуются через Foreign Key.

Связи без ограничений не допускаются.

---

# Удаление данных

Физическое удаление используется только в исключительных случаях.

По возможности применяется Soft Delete.

---

# История изменений

Все изменения записываются в Audit Log.

История не изменяется.

История не удаляется.

---

# Timeline

Timeline отражает историю жизни сообщества.

Каждая запись является событием.

Timeline не является журналом изменений данных.

Для этого используется Audit Log.

---

# AI

AI не изменяет таблицы пользователей напрямую.

AI работает только через собственные сущности.

Это позволяет переанализировать данные без изменения фактов.

---

# Масштабируемость

Архитектура должна поддерживать:

- несколько сообществ;
- несколько Telegram-групп;
- несколько AI-моделей;
- несколько источников данных;
- несколько внешних сервисов.

Без изменения существующей структуры базы данных.

# Таблица: users

## Назначение

Хранит основную информацию о пользователе системы.

Пользователь является центральной сущностью платформы.

Все остальные данные связываются с пользователем через внешний ключ.

---

## Ответственность

Таблица хранит только фактические данные.

В ней не должно быть:

- AI-анализа;
- социальных сетей;
- интересов;
- навыков;
- рекомендаций.

Эти данные располагаются в специализированных таблицах.

---

## Поля

| Поле | Тип | Описание |
|------|-----|----------|
| id | BIGSERIAL | Первичный ключ |
| telegram_id | BIGINT UNIQUE | Telegram ID пользователя |
| username | VARCHAR(255) | Username Telegram |
| first_name | VARCHAR(255) | Имя |
| last_name | VARCHAR(255) | Фамилия |
| language_code | VARCHAR(10) | Язык Telegram |
| is_bot | BOOLEAN | Telegram Bot |
| is_premium | BOOLEAN | Telegram Premium |
| is_member | BOOLEAN | Находится в сообществе |
| is_admin | BOOLEAN | Администратор |
| is_banned | BOOLEAN | Заблокирован |
| joined_at | TIMESTAMP | Дата вступления |
| left_at | TIMESTAMP NULL | Дата выхода |
| first_seen_at | TIMESTAMP | Первое появление |
| last_seen_at | TIMESTAMP | Последнее появление |
| last_activity_at | TIMESTAMP | Последняя активность |
| last_message_at | TIMESTAMP | Последнее сообщение |
| messages_count | INTEGER | Количество сообщений |
| media_count | INTEGER | Количество медиа |
| replies_count | INTEGER | Количество ответов |
| created_at | TIMESTAMP | Создание записи |
| updated_at | TIMESTAMP | Последнее обновление |

---

## Индексы

Создаются индексы:

- telegram_id (UNIQUE)
- username
- is_member
- last_activity_at
- last_message_at

---

## Связи

User имеет:

- один Profile;
- один AIProfile;
- много Messages;
- много Timeline событий;
- много Recommendations;
- много Organizations;
- много Projects;
- много EventParticipants;
- много Achievements.

---

## Ограничения

Telegram ID является уникальным.

Удаление пользователя не должно удалять историю сообщества.

По возможности используется Soft Delete.

---

## Примечания

Таблица users должна оставаться компактной.

Не допускается добавление в неё данных, относящихся к другим сущностям.

При появлении новой функциональности создаётся новая таблица, а не новые десятки колонок.
