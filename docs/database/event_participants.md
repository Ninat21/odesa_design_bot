# Таблиця `event_participants`

## Призначення

Таблиця `event_participants` зберігає інформацію про участь користувачів у заходах.

Вона реалізує зв'язок «багато до багатьох» між таблицями `users` та `events`, а також містить службову інформацію про реєстрацію, статус участі та отримані згоди.

---

# Відповідальність

Таблиця відповідає за:

- реєстрацію користувача на захід;
- статус участі;
- спосіб реєстрації;
- фіксацію отриманих згод;
- інформацію про підтвердження присутності.

---

# Поля

| Поле | Тип | NULL | Опис |
|------|-----|------|------|
| id | BIGSERIAL | ❌ | Первинний ключ |
| event_id | BIGINT | ❌ | Посилання на `events.id` |
| user_id | BIGINT | ❌ | Посилання на `users.id` |
| external_participant_id | VARCHAR(255) | ✅ | ID учасника у зовнішній системі (наприклад, Luma) |
| registration_method | VARCHAR(30) | ❌ | Спосіб реєстрації |
| status | VARCHAR(30) | ❌ | Поточний статус участі |
| consent_personal_data | BOOLEAN | ❌ | Згода на обробку персональних даних |
| consent_marketing | BOOLEAN | ❌ | Згода на отримання інформації про майбутні заходи |
| consent_photo | BOOLEAN | ❌ | Згода на фото- та відеозйомку |
| checked_in_at | TIMESTAMP | ✅ | Час підтвердження присутності |
| checked_in_by | BIGINT | ✅ | Користувач, який підтвердив присутність |
| ticket_type | VARCHAR(100) | ✅ | Тип квитка |
| registered_at | TIMESTAMP | ✅ | Дата реєстрації |
| cancelled_at | TIMESTAMP | ✅ | Дата скасування реєстрації |
| created_at | TIMESTAMP | ❌ | Дата створення запису |
| updated_at | TIMESTAMP | ❌ | Дата останнього оновлення |

---

# Первинний ключ

```
id
```

---

# Зовнішні ключі

```
event_id → events.id

user_id → users.id

checked_in_by → users.id
```

---

# Обмеження

Один користувач може мати лише одну реєстрацію на один захід.

```
UNIQUE(event_id, user_id)
```

# Згоди

Для кожної реєстрації зберігається стан усіх отриманих згод.

Це дозволяє:

- підтвердити факт отримання згоди;
- враховувати різні умови для різних заходів;
- зберігати історичний стан згод незалежно від подальших змін у профілі користувача.

# Правила оновлення

Статус участі може змінюватися протягом життєвого циклу заходу.

Після завершення заходу інформація зберігається як історична.

# Причини такої структури

Таблиця реалізує зв'язок «багато до багатьох» між користувачами та заходами.

Зберігання статусу участі, способу реєстрації та згод в окремій таблиці дозволяє вести повну історію участі користувача в заходах без дублювання інформації.
