# Таблиця `user_organizations`

## Призначення

Таблиця зберігає історію роботи, навчання та інших зв'язків користувача з організаціями.

Один користувач може мати необмежену кількість записів.

---

# Відповідальність

Таблиця описує конкретний зв'язок між користувачем та організацією.

---

# Поля

| Поле | Тип | NULL | Опис |
|------|-----|------|------|
| id | BIGSERIAL | ❌ | Первинний ключ |
| user_id | BIGINT | ❌ | users.id |
| organization_id | BIGINT | ❌ | organizations.id |
| position | VARCHAR(255) | ✅ | Посада |
| department | VARCHAR(255) | ✅ | Відділ |
| employment_type | VARCHAR(100) | ✅ | Тип зайнятості |
| started_at | DATE | ✅ | Початок роботи |
| ended_at | DATE | ✅ | Завершення роботи |
| is_current | BOOLEAN | ❌ | Поточне місце роботи |
| description | TEXT | ✅ | Коментар |
| source | VARCHAR(100) | ✅ | Джерело інформації |
| created_at | TIMESTAMP | ❌ | Створення запису |
| updated_at | TIMESTAMP | ❌ | Останнє оновлення |

# Правила оновлення

Історія роботи не повинна видалятися.

При зміні місця роботи попередній запис завершується встановленням `ended_at`.

---

# Джерела даних

- Користувач
- Адміністратор
- Імпорт
- AI (лише як пропозиція, без автоматичного запису)

---

# Причини такої структури

Користувач може працювати:

- у декількох компаніях;
- на декількох посадах;
- у різні періоди часу.

Зберігання лише одного поля `company` у профілі не дозволяє відобразити професійну історію.

---

# Майбутній розвиток

Можливе додавання:

- рівня посади;
- формату роботи (офіс, віддалено, гібрид);
- зарплатного діапазону (за бажанням користувача);
- посилання на вакансію або проєкт.
