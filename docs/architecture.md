# Architecture

> Архитектура Odesa Community Platform

Version: 1.0

---

# Общая архитектура

Платформа построена по модульному принципу.

Каждый модуль отвечает только за одну область ответственности и взаимодействует с остальными через общую бизнес-логику.

```
                         User

                           │

          ┌────────────────┼────────────────┐

          │                │                │

      Telegram         Website         Admin Panel

          │                │                │

          └────────────────┴────────────────┘

                           │

                     Application Layer

                           │

      ┌─────────────┬──────────────┬─────────────┐

      │             │              │             │

 Community      AI Engine     Analytics      Integrations

      │             │              │             │

      └─────────────┴──────────────┴─────────────┘

                           │

                     PostgreSQL

                           │

      ┌─────────────┬──────────────┬─────────────┐

      │             │              │

  Telethon       OpenAI         Luma API
```

# Основные компоненты

## Presentation Layer

Отвечает за взаимодействие с пользователем.

Включает:

- Telegram Bot
- Website
- Admin Panel
- будущие мобильные приложения

Эти компоненты не содержат бизнес-логики.

---

## Application Layer

Главный уровень системы.

Именно здесь реализуется:

- бизнес-логика
- проверка прав
- сценарии работы
- маршрутизация данных
- взаимодействие между сервисами

Ни один интерфейс не обращается напрямую к базе данных.

---

## Database Layer

Единый источник данных.

Используется PostgreSQL.

База данных не зависит от Telegram или любого другого сервиса.

---

# Основные модули

## Identity

Отвечает за пользователей.

Содержит:

- пользователей
- аккаунты
- профили
- контакты
- историю изменений

---

## Community

Работает с сообщениями.

Содержит:

- сообщения
- вложения
- реакции
- упоминания
- ссылки

---

## Events

Работает с мероприятиями.

Поддерживает различные платформы.

Например:

- Luma
- собственные мероприятия
- будущие интеграции

---

## AI

Полностью независимый модуль.

Отвечает за:

- анализ сообщений
- анализ пользователей
- построение AI-профилей
- рекомендации
- поиск
- извлечение знаний

AI не изменяет фактические данные.

---

## Knowledge

Коллективная память сообщества.

Включает:

- статьи
- FAQ
- лучшие практики
- выводы AI
- связи с сообщениями

---

## Analytics

Вычисляет статистику.

Например:

- активность
- рост сообщества
- вовлеченность
- посещаемость мероприятий
- популярные темы

Analytics не изменяет данные.

---

## Notifications

Отправляет сообщения пользователям.

Источниками могут быть:

- Telegram
- Email
- Website

# Архитектурные принципы

## Single Source of Truth

Все данные находятся только в PostgreSQL.

---

## Separation of Concerns

Каждый модуль отвечает только за одну область ответственности.

---

## Event Driven

Любое значимое действие является событием.

События сохраняются в Timeline.

---

## Audit First

Любое изменение данных фиксируется в Audit Log.

---

## AI Isolation

AI работает только со своими сущностями.

Он не изменяет фактические данные напрямую.

---

## Scalability

Любой модуль может быть заменён без изменения остальных частей системы.

# Репозитории

Каждая сущность имеет собственный Repository.

Например:

- UserRepository
- MessageRepository
- EventRepository
- KnowledgeRepository

Repository отвечает только за работу с базой данных.

---

# Сервисы

Сервисы содержат бизнес-логику.

Например:

- AIService
- AnalyticsService
- LumaService
- TelegramImportService
- KnowledgeService

---

# Принципы зависимостей

Handlers не работают напрямую с базой данных.

Handlers работают только через Services.

Services используют Repositories.

Repositories работают с SQLAlchemy.

---

# Основная цель архитектуры

Любой модуль системы может развиваться независимо, не нарушая работу остальных компонентов.
